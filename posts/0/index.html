#【创新APP】Spring、Typhone、Firebase 等框架考察
[TOC]

前言

针对创新APP框架中的Service层的设计，研究了几款 IOC 框架（Service框架）。并尝试对两款 IOC 框架的 生命周期管理、循环引用处理、配置方式等能力做了对比分析（见本文“对比能力”部分）。希望对 Service 的设计有所帮助。
周一(12.10日)上午晨会时，已经和 haven、kelvin 、heysea 同步了研究分析的结果，并确定如下4想需要细化的内容：

  1. 接口的自动类型转换。
  2. 循环引用的处理办法。
  3. 服务创建过程中的多线程管理。
  4. 服务生命周期管理。

下表是暂阶段我们的service框架（ours）与主流框架能力的对比。

主要功能

spring IOC， AOP
typhone IOC

配置方式

spring：
重配置，轻编译
xml，anotation， java code

typhone:
重编译（编译前类型检测），轻配置（只能注入属性）
java code, (json,plist 只支持属性）

anotation 注入

spring：支持
typhone：不支持，由于直接采用oc和swift代码，因此也不需要anotation

多context的支持

spring： 支持
typnone： 通过let assembly = LoginAssembly().activate() 即实例化不同的"类厂"来支持

compile-time checking

spring：不支持
typnone： 支持
